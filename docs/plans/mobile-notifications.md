## 웹 계획서에서 재사용하는 분석 (다시 반복하지 않음)
아래 내용은 `docs/plans/notifications.md`에 이미 정리되어 있고 이번에도 그대로 유효하다 (다시 조사하지 않고 인용만 함):
- 피그마 파일은 "화면 목업 + 스펙 설명 주석"이 절대 좌표로만 배치된 플랫 캔버스 구조다.
- 검은 원(`Ellipse 9`, 32×32) + 숫자 텍스트 조합(`Group N`)은 **여전히 실제 UI가 아니다** → 화면 구성에서 제외. 다만 이번 이슈 목적상 **그 마커가 가리키는 설명 텍스트는 전부 찾아 요구사항으로 반영**했다 (아래 "마커별 요구사항 대응표" 참고).
- 알림 아이템 콘텐츠 스펙(고민명 작은따옴표 표기, 메시지, 상대시간, 안읽음 dot, 썸네일)은 앱/웹 동일 — 기존 `NotificationListItem`/`NotificationItem` 타입을 그대로 재사용한다.
- 기기 프레임(iOS 상태바 `System/iOS/StatusBar`, 노치, 홈 인디케이터 `System/iOS/HomeIndicator`)은 이번에도 전부 구현 대상에서 제외한다.

## 빨강 vs 초록 판단 근거 (사용자 지시 2 대응)

개별 삭제 스와이프 화면은 피그마에 두 벌 존재한다. 색상을 직접 조회해 확인했다.

- `127:373`("알림 화면 - 개별 삭제") → 스와이프 시 나타나는 삭제 버튼 배경 `127:435`가 **`bg-[#a9d592]` (초록, 기존 안읽음 dot과 동일 톤)**
- `130:202`("알림 화면 - 개별 삭제") → 스와이프 시 나타나는 삭제 버튼 배경 `130:230`가 **`bg-[#ec2d30]` (빨강)**
- 디자이너 메모(`130:253`)가 "둘 다 만들어 뒀으니 알아서 정하라"고 명시 → **사용자 지시에 따라 `130:202`(빨강, `#ec2d30`) 쪽 스펙을 채택**한다. `127:373`(초록) 쪽은 참고용으로만 남기고 구현하지 않는다.
- 빨강 버튼 세부 실측(`130:202` 기준): 배경 `130:230` 40×35px, 라운드 없음(각진 사각형), 그 위에 흰색 휴지통 아이콘(`130:231`, "image 138", 26×28, 위치상 버튼 중앙 정렬). 아이템 로우 높이는 대략 56px(썸네일 40px + 상하 패딩), 스와이프 시 아이템 콘텐츠가 왼쪽으로 약 40px 밀리며 오른쪽에 이 버튼이 드러난다.
- 참고로 오버플로우 메뉴의 "알림 전체 삭제" 텍스트 색상도 동일한 `#ec2d30`라, 이 프로젝트의 "삭제" 관련 액션 색상을 빨강(`#ec2d30`)으로 일관되게 확정할 수 있다.

## 피그마 실측 세부 스펙

### 헤더(더보기 버튼)
- 모든 화면 공통: 우측 상단에 케밥(⋮) 아이콘 하나만 존재 (라벨/뒤로가기 텍스트 없음). 아이콘은 세로 점 3개(`image 99` 등, 29×29) — 기존 웹 드롭다운에서 이미 쓰는 `lucide-react`의 `MoreVertical`과 시각적으로 동일. 새 에셋 불필요.
- 기존 `NotificationList`의 더보기 버튼(`disabled` 처리된 것)을 모바일 페이지에서는 활성화하고 클릭 시 오버플로우 메뉴를 띄운다.

### 오버플로우 메뉴 (`126:136`)
- 배경 `bg-[#f5f5f5]`, `rounded-[6px]`, `shadow-[0px_2px_4px_0px_rgba(0,0,0,0.04),0px_1px_1px_0px_rgba(0,0,0,0.02)]`, 154×64px, 케밥 버튼 바로 아래 우측 정렬로 나타남(피그마상 케밥 아이콘 우측 끝과 메뉴 우측 끝이 거의 일치).
- 아이템 1: "모두 읽음으로 표시" — `text-black`, 13px, 일반 굵기.
- 아이템 2: "알림 전체 삭제" — `text-[#ec2d30]`, 13px.
- 두 아이템 사이 구분선은 피그마에서 발견되지 않음(단순히 위아래로 쌓인 텍스트) — 확인 필요 목록 참고.

### 전체 삭제 확인 모달 (`127:160` 관련)
- 모달 박스(`127:225`): `bg-white`, `rounded-[6px]`, `shadow-[0px_2px_4px_0px_rgba(0,0,0,0.04),0px_1px_1px_0px_rgba(0,0,0,0.02)]`, 260×100px, 화면 중앙.
- 문구(`127:227`): "알림을 모두 삭제하시겠습니까?" — `font-bold`, 15px, `text-black`.
- 버튼 2개, 각각 98×24px, `bg-[#eefff0]`, `rounded-[6px]`:
  - "네"(`127:232`) — `font-medium`, 14px, `text-black` → 확인 시 전체 삭제(`deleteAll`) 실행 후 모달 닫힘.
  - "아니요"(`127:236`) — 동일 스타일 → 모달만 닫고 목록은 그대로.

### 개별 삭제 확인 모달 (`130:139` 관련)
- 모달 박스/버튼 스타일은 전체 삭제 모달과 완전히 동일(같은 색/크기/라운드).
- 문구(`130:175`): "해당 알림을 삭제하시겠습니까?" — `font-bold`, 15px, `text-black`.
- "네" → 해당 알림 1건만 삭제(`deleteNotification(id)`), 스와이프 상태 초기화, 모달 닫힘.
- "아니요" → 모달만 닫고, 해당 아이템은 "스와이프 전" 상태로 복구(디자이너 주석에 명시된 대로).

### 개별 삭제 스와이프 (빨강 변형 채택, `130:202` 관련)
- 왼쪽으로 스와이프(오→왼) → 아이템이 왼쪽으로 이동하며 오른쪽에 빨간 삭제 버튼(위 실측값) 노출.
- 삭제 버튼 탭 → 개별 삭제 확인 모달 오픈.
- 다시 오른쪽으로 스와이프(왼→오) → 버튼이 사라지고 원상 복구.
- 마우스 환경(데스크톱 폭에서 우연히 접근한 경우 등)을 고려해 스와이프 대신 "탭하면 버튼이 나타나고 다시 탭하면 닫히는" 폴백도 고려 가능 — 정확한 폴백 방식은 확인 필요 목록 참고.

## 화면 구성

### 화면 0: `NotificationBell` 모바일 탭 동작 변경 (트리거)
- 기존 `src/components/NotificationBell/index.tsx`는 클릭 시 드롭다운(`NotificationsPanel`)을 토글하는 로직만 있다. 이 컴포넌트는 `MobileTopBar`(모바일)와 `HomeHeader`(데스크톱) 두 곳에 그대로 재사용되고 있다.
- **데스크톱(≥ `lg`)**: 기존과 동일하게 벨 클릭 → 드롭다운 패널 토글 (변경 없음, 이슈 #15 완료 스펙 유지).
- **모바일(< `lg`)**: 피그마 모바일 목업이 작은 팝오버가 아니라 **상태바/케밥 버튼만 있는 완전한 풀스크린**이므로, 벨 탭 → `/notifications` 페이지로 라우팅 이동(드롭다운 아님).
- 구현 방식 제안: `NotificationBell`에 `variant?: 'dropdown' | 'page'` 같은 prop을 추가하고, `MobileTopBar`에서 `variant="page"`로 넘겨 클릭 시 `useNavigate()`로 `ROUTES.notifications` 이동, `HomeHeader`는 prop 생략(기본값 `dropdown`) 유지. (다른 방식으로 구현해도 되나, 두 마운트 지점이 이미 있다는 점은 고려해야 함.)

## 상태/데이터 흐름 변경 (`NotificationsContext`)

`src/contexts/NotificationsContext.tsx`에 아래 액션을 추가해야 한다 (타입/더미 데이터 자체는 변경 없음):

- `deleteNotification(id: string): void` — 배열에서 해당 id 제거
- `deleteAll(): void` — 배열을 빈 배열로 교체
- `markAllAsRead(): void` — 모든 아이템의 `isRead`를 `true`로 일괄 변경 (기존 `markAsRead`는 단건용으로 유지)

이 세 액션 모두 클라이언트 상태만 변경하면 되고(더미 데이터, 새로고침 시 초기화되어도 무방 — 이슈 #15 확인 완료 사항과 동일한 전제), Supabase 연동은 이번 이슈 범위가 아니다.

## 라우팅 구조 관련 중요 확인 필요 (설계 결정)

피그마의 알림 관련 모바일 목업(`92:53`, `151:158`, `126:73`, `127:160`, `127:373`/`130:202`, `130:139` 전부)을 살펴보면, 상단에는 **케밥 버튼만 있는 얇은 헤더**만 있고 로고+벨이 있는 기존 `MobileTopBar`나 하단 `BottomNav` 탭바가 전혀 그려져 있지 않다(리스트 아래로 상당한 빈 흰 여백만 있고 별도 요소 없음). 반면 현재 프로젝트는 `AppLayout`이 모든 페이지에 `MobileTopBar`+`BottomNav`를 항상 씌운다.

두 가지 구현 방향이 가능하며, 이는 코드 구조에 영향을 주므로 **반드시 사용자 확인 후 진행**해야 한다 (아래 "확인 필요" 참고):
1. **피그마 그대로**: `/notifications`를 `AppLayout` 밖에 별도 라우트로 등록(로그인/회원가입 페이지처럼)해 자체적인 흰 배경 풀스크린으로 구현. 이 경우 페이지를 벗어날 방법(뒤로가기 버튼 등)이 피그마에 없으므로 별도로 필요.
2. **기존 셸 유지**: `AppLayout` 안에서 라우팅하되 이 라우트일 때만 `MobileTopBar`/`BottomNav`를 숨기는 조건부 처리, 혹은 아예 유지(피그마와 다르지만 앱 내비게이션 일관성 확보).

## 확인 완료 (사용자 승인)

- **라우팅 셸**: 피그마 그대로 `AppLayout` 밖에 독립 풀스크린 라우트로 구현 (로그인/회원가입 페이지와 동일한 패턴).
- **뒤로가기/닫기 수단**: 피그마엔 없지만 사용성을 위해 헤더 케밥 버튼 옆에 뒤로가기 아이콘을 추가한다 (`lucide-react` `ChevronLeft` 등, 클릭 시 이전 화면으로 이동 — `navigate(-1)` 또는 홈으로 고정, 구현 시 자연스러운 쪽으로).
- **`lg`(425px) 이상 접근 시**: 홈(`/`)으로 리다이렉트한다.
- **아이템 클릭 시 동작**: `markAsRead`만 호출, 이동 없음 (웹과 동일).
- **스와이프 동시 오픈 정책**: 한 번에 하나만 열림 — 다른 아이템을 스와이프하면 이전에 열려 있던 아이템은 자동으로 닫힌다 (iOS 패턴).
- **마우스 폴백**: 터치 스와이프뿐 아니라 클릭으로도 토글 가능하게 구현한다 (데스크톱 브라우저에서 접근했을 때도 삭제 버튼을 열고 닫을 수 있어야 함).
- **휴지통 아이콘**: `lucide-react`의 `Trash2`로 대체 (흰색).
- **빈 상태 케밥 버튼**: 비활성화 상태로 그대로 둔다 (숨기지 않음).
- **모달 딤(dim) 배경**: `bg-black/40`로 진행 (표준값, 낮은 리스크 사항이라 별도 확인 없이 확정).
- **오버플로우 메뉴 구분선**: 피그마에 없는 대로 구분선 없이 구현한다.
