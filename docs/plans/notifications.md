# 웹 알림 기능 구현 계획

## 개요
- 피그마 링크: https://www.figma.com/design/Tjb8LmHOOZXQMLSL86LSJC/일상뒤집기-디자인?node-id=0-1 (fileKey `Tjb8LmHOOZXQMLSL86LSJC`)
- 관련 화면: `/notifications` 페이지 (홈 화면 `NotificationBell` 클릭 시 이동하는, 아직 없는 실제 화면)
- 한 줄 요약: 알림 목록(무한 스크롤 리스트) + 빈 상태 2가지 화면을 구현한다. **피그마에 명시된 주석에 따라 삭제/전체삭제/모두읽기 관련 UI는 웹 스코프에서 전부 제외**한다.

## 피그마 구조 분석 메모 (중요 — 개발 시 반드시 참고)

이 피그마 파일은 "화면 목업 + 스펙 설명 주석"이 하나의 캔버스에 좌표로만 배치된 플랫(flat) 구조다(제대로 중첩된 프레임이 아님). 그래서 아래처럼 영역별로 특정해서 분석했다.

1. **모바일 앱 목업 (x≈426~1252, y≈-380~432)**: `알림 화면 - 기본`(92:53) 등 여러 상태의 iOS 앱 목업. 상태바/노치/홈 인디케이터 등 기기 프레임은 프로젝트 규칙에 따라 전부 제외.
2. **웹 목업 (x≈1971~2338, y≈2052~2529, 배경 노드 130:359)**: 홈 화면 우상단 벨 아이콘(image147, 36×36, 이미 구현된 `NotificationBell`과 동일 위치) 클릭 시 나타나는 것으로 보이는 **367×477px 흰색 드롭다운 패널**. 이것이 "웹 알림 화면"의 실질적인 비주얼 원본이다.
   - 패널 스타일: `bg-white`, `border` `#e1e1e1` 0.5px, `rounded-[20px]`, `shadow-[0px_2px_4px_rgba(0,0,0,0.08),0px_1px_1px_rgba(0,0,0,0.04)]`.
   - 이 패널은 앱 목업의 "알림 화면 - 기본"(92:53)과 **동일한 5개 알림 아이템**(같은 문구·같은 순서·같은 시간·같은 안읽음 상태)을 담고 있어, 콘텐츠 스펙은 모바일/웹이 동일함을 확인했다.
3. **디자이너 주석(스펙 설명용 텍스트박스)** 중 결정적인 문구(원문 그대로):
   > "알림창은 무한 스크롤입니다. 알림 클릭 시 해당 타이머로 이동합니다. **웹의 경우 알림 삭제하기 기능, 모두 읽기 기능은 없습니다.** 그 밖에 내용은 앱과 동일합니다."
   - 즉 **무한 스크롤 / 클릭 시 타이머 이동은 웹에도 적용**되지만, **더보기(⋮) 오버플로우 메뉴("모두 읽음으로 표시" / "알림 전체 삭제"), 전체삭제 확인 모달, 개별 알림 스와이프 삭제 + 확인 모달은 앱 전용 기능이며 웹에는 없다.** 아래 "제외된 요소" 섹션 참고.
4. **사용자가 사전에 알려준 대로**, "검은 원 안에 숫자(1, 2, 3…)" 마커(`Ellipse 9` 32×32 원 + 숫자 텍스트가 짝을 이루는 `Group N` 형태, 예: 101:340/341/342, 92:111/112/113 등)는 전부 스펙 설명용 주석이며 **실제 UI 아님 → 컴포넌트 트리/화면 구성에서 전부 제외**했다.

## 화면 구성

### 화면 1: 알림 목록 (기본 — 알림 있음)
- 라우트: `/notifications` (`ROUTES.notifications`, 이미 `src/routes/paths.ts`에 정의돼 있음)
- 컴포넌트 트리 (제안):
  - `NotificationsPage` (`src/pages/Notifications/index.tsx`)
    - `NotificationList`
      - `NotificationListItem` (반복) — 아이콘/썸네일, 제목(`'치즈 말랑이'` 형태로 작은따옴표 포함), 설명, 상대시간, 안읽음 표시 dot
    - (무한 스크롤 트리거: 리스트 하단 sentinel 또는 스크롤 컨테이너 onScroll)
  - 기존 `AppLayout`(Sidebar/MobileTopBar/BottomNav) 안에서 라우팅되는 일반 페이지로 구현 (기존 `/` 홈과 동일한 레이아웃 셸 재사용)
- 상태/데이터:
  - 알림 리스트 아이템 필드(피그마 실측 기준): `id`, `worryName`("'치즈 말랑이'" 처럼 작은따옴표로 감싼 고민명), `message`(설명 텍스트), `createdAt`/상대시간 표시("13시간 전" 등), `isRead`(boolean — 안읽음이면 좌측에 연두색 dot `#a9d592` 8×8), `thumbnailUrl`(고민 썸네일, 없으면 `WorryListItem`과 동일하게 placeholder 박스 처리)
  - 정렬: 최신순 (맨 위 = 최신 알림, 맨 아래 = 가장 오래된 알림)
  - 페이지네이션: 무한 스크롤 (정확한 페이지 크기/커서 전략은 "확인 필요" 참고)
- 인터랙션:
  - 알림 아이템 클릭 → "해당 타이머로 이동" (구체적 목적지는 "확인 필요" 참고 — 현재 프로젝트에 고민 상세/타이머 라우트가 아직 없음)
  - 더보기(⋮) 메뉴, 전체삭제, 개별삭제(스와이프) — **웹 스코프에서 구현하지 않음** (아래 "제외된 요소" 참고)
- 필요 에셋/아이콘:
  - 아이템 썸네일: `worry`의 `thumbnailUrl` 재사용 (홈 화면 `OngoingWorry.thumbnailUrl`과 동일 패턴, 없으면 회색 placeholder 박스)
  - 안읽음 표시: 별도 아이콘 없이 `bg-[#a9d592]` 원(8×8, Tailwind로는 `h-2 w-2` 근접값)으로 구현 — 기존 프로젝트에서 이미 쓰는 `#a9d592` 그린 토큰과 동일 색상
  - `src/assets/alarm-exist.svg`: 이미 제공된 에셋. 벨+연두 점 아이콘으로, 홈 화면 벨(`NotificationBell`, `alarm.svg` 고정 사용은 이전 이슈에서 "확인 완료"로 이미 결정된 사항이라 변경하지 않음)과는 별개로, 이 알림 목록 화면 자체의 어딘가(예: 빈 아이템이 없을 때와 대비되는 상단 아이콘 등)에 쓰일 가능성이 있는 것으로 추정되나, 피그마상 정확한 배치 위치를 특정하지 못했다 → "확인 필요" 참고
  - lucide-react: 필요 시 뒤로가기 등 헤더 아이콘 (피그마에는 이 화면 헤더에 타이틀/뒤로가기 텍스트·아이콘이 없어 확인 필요)
- 반응형 기준:
  - 프로젝트 기존 브레이크포인트(`lg`=425px, `xl`=1024px) 그대로 사용
  - 피그마 웹 목업은 367px 폭의 작은 드롭다운 패널이지만, 라우팅은 이미 전체 페이지 이동(`/notifications`)으로 고정되어 있으므로, 리스트는 페이지 콘텐츠 영역 폭에 맞춰 유동적으로 늘어나는 카드/리스트로 구현하고 아이템 내부 비율(아이콘 40px, 텍스트 크기 등)은 피그마 실측값을 기준으로 유지한다. 폭 처리 방식은 "확인 필요"에서 최종 확정.
  - 모바일 폭에서는 `AppLayout`의 기존 모바일 톱바/하단 네비게이션 구조를 그대로 사용 (기기 프레임인 상태바/노치/홈 인디케이터는 구현 대상 아님 — 실제 브라우저가 담당)

### 화면 2: 알림 목록 (빈 상태)
- 라우트: 동일 `/notifications` (알림이 0개일 때 조건부 렌더링, 별도 라우트 아님)
- 컴포넌트 트리:
  - `NotificationsPage`
    - `NotificationEmptyState` — 일러스트 아이콘 + "아직 알림이 없어요" 텍스트
- 상태/데이터: 알림 개수 0건일 때 표시
- 인터랙션: 없음
- 필요 에셋/아이콘:
  - 빈 상태 일러스트(가방+시계+체크 모양 커스텀 아이콘, 피그마 노드 151:221/130:279): **현재 프로젝트에 해당 에셋이 없다.** `alarm-exist.svg`는 벨 아이콘이라 이 용도가 아님 → 새 에셋 필요 (확인 필요 참고)
  - 텍스트: "아직 알림이 없어요" (15px, `font-medium`, `text-black`)
- 반응형 기준: 페이지 중앙 정렬, 폭에 관계없이 콘텐츠는 고정 크기로 중앙에 위치(피그마상 리스트처럼 늘어나는 요소가 아님)

## 확인 완료 (사용자 승인)

- **데이터 소스**: 이번 이슈에서는 더미 데이터로 구현한다. Supabase `notifications` 테이블 등 실제 연동은 별도 이슈로 미룬다.
- **알림 클릭 시 동작**: 다른 화면/라우트로 이동하지 않는다. 웹에서는 "읽기"만 가능하므로, 클릭하면 해당 알림의 `isRead`를 `true`로 바꿔 안읽음 dot을 제거하는 동작만 한다 (네비게이션 없음).
- **`alarm-exist.svg` 사용 위치**: 알림이 있는 상태의 리스트에서, 각 `NotificationListItem`의 왼쪽 아이콘으로 사용한다 (아이템마다 이 벨 아이콘을 표시).
- **빈 상태 일러스트**: 새 에셋을 만들지 않고 기존 `src/assets/electric_bulb.svg` 등으로 대체한다.
- **웹 리스트 폭**: 피그마 카드 폭(367px)에 가깝게 좁고 중앙 정렬된 리스트로 구현한다 (페이지 전체 폭으로 늘리지 않음).
- **헤더**: 피그마대로 타이틀/뒤로가기 없이 리스트만 보여준다.
- **무한 스크롤**: 이번엔 더미 데이터 5개만 간단히 렌더링한다. 실제 무한 스크롤(페이지네이션/커서) 로직은 나중에 Supabase 연동할 때 함께 구현한다 — 이번엔 스크롤 트리거 로직 자체를 만들 필요 없다.
- **더보기(⋮) 버튼**: 버튼은 두되 비활성화 처리한다 (클릭 안 되게, 시각적 일관성 유지).

## 작업 순서 제안 (developer 서브에이전트용)

1. `src/routes/paths.ts`의 `notifications` 경로를 사용해 `src/pages/Notifications/index.tsx` 페이지 골격 생성, `App.tsx`의 `ProtectedRoute`+`AppLayout` 하위에 라우트 추가.
2. 알림 아이템 타입(`src/types/notifications.ts` 등) 정의 — "확인 필요"의 데이터 소스 답변에 맞춰 더미 데이터 또는 Supabase 연동 훅(`useNotificationsData` 등, `useHomeData` 패턴 참고) 작성.
3. `NotificationListItem` 컴포넌트 구현 (아이콘/썸네일 + 제목 + 설명 + 상대시간 + 안읽음 dot), 피그마 실측 타이포(제목 11px semibold, 설명/시간 9px medium `#666`) 반영, 프로젝트 기본 폰트 스택 사용(피그마의 Inter/Pretendard 폰트 지정은 무시, 기존 프로젝트 컨벤션 따름).
4. `NotificationList` 컴포넌트로 리스트 렌더 + 무한 스크롤 처리.
5. `NotificationEmptyState` 컴포넌트 구현 (일러스트 에셋 확정 후).
6. 알림 0건/1건 이상 분기 렌더링을 `NotificationsPage`에서 처리.
7. 반응형 점검: `lg`(425px)/`xl`(1024px) 기준 폭 변화에서 리스트 레이아웃 확인.
8. "확인 필요" 항목들은 반드시 AskUserQuestion으로 확인 후 구현 — 임의 판단 금지.
