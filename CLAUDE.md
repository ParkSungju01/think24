# 일상 뒤집기 공모전 프로젝트

## 개요
- 이 저장소는 "일상 뒤집기" 공모전 제출작이다.
- 스택: Vite + React 19 + TypeScript, Tailwind CSS, react-router-dom(멀티 페이지), lucide-react(아이콘)
  - Tailwind / react-router-dom / lucide-react는 아직 설치되어 있지 않다. 개발 단계에서 처음 필요할 때 설치한다.
- 디자인 소스: Figma 링크 (작업 시작 시 사용자가 전달)

## 작업 원칙
- 디자인은 전달받은 Figma 그대로 구현한다. 레이아웃/색상/타이포/간격을 임의로 바꾸지 않는다.
- 에셋은 사용자가 미리 넣어주거나(예: `src/assets`, `public/assets`) lucide-react 아이콘을 사용한다. 계획에 없는 에셋을 새로 만들지 않는다.

## 브랜치 전략
- `main`: 배포 브랜치. **오직 이 브랜치만 배포한다.** 서브에이전트는 이 브랜치에 직접 커밋/머지하지 않는다.
- `develop`: 통합 브랜치. 모든 기능 브랜치는 이 브랜치로 PR을 올린다.
- 기능 브랜치는 **사용자가 이슈를 만들면서 함께 생성**해둔다(`develop`에서 분기, GitHub 기본 규칙상 보통 `<이슈번호>-<slug>` 형태). developer 서브에이전트는 새 브랜치를 만들지 않고, 해당 이슈에 연결된 기존 브랜치를 checkout해서 작업한다.
- `develop → main` 승격(배포 시점 결정)은 파이프라인 밖에서 사용자가 직접 진행한다.

## 파이프라인: 기획 → (사용자 승인) → 개발 → 검토 → (재작업 루프) → PR 대기

```
사용자: 피그마 링크 전달
   │
   ▼
[planner 서브에이전트] 기획
   - 피그마 분석, docs/plans/<slug>.md 작성
   - 코드 수정 금지 (계획 문서만 작성)
   │
   ▼
사용자 승인 게이트 ── 계획 확인/수정 요청
   │
   ▼
[developer 서브에이전트] 개발
   - 사용자가 이슈 생성 시 만들어둔 기존 브랜치를 checkout (새 브랜치 생성 안 함)
   - 계획 기반 구현, 기능 단위로 작업
   - 애매한 부분(계획에 없거나 여러 해석 가능) → 임의 판단 금지, AskUserQuestion으로 확인
   - 기능 완료마다: 변경사항 + 로직 설명 → 커밋 메시지 제안 → 사용자 승인 후 커밋·push
   - 모든 기능 구현 후 develop 대상 PR 생성 (PR 템플릿 사용, 승인 없이 바로 진행)
   │
   ▼
[reviewer 서브에이전트] 검토
   - 빌드/린트/타입체크, 라우팅, 피그마 대비 시각 비교, 반응형 점검
   - 이슈 발견 시 developer 서브에이전트에게 자동으로 재작업 요청 (사용자 개입 없이 루프)
     - 재작업 커밋도 동일하게 사용자 승인 후 커밋·push, PR 갱신은 승인 없이 진행
   - 이슈가 없을 때까지 반복 후 최종 결과를 사용자에게 보고
   │
   ▼
PR은 올라간 상태로 대기 — develop 머지는 항상 사용자가 직접 진행
```

- 각 화살표(→)는 상위 대화(오케스트레이터)에서 Agent 도구로 해당 서브에이전트를 호출하는 지점이다.
- **사용자 승인이 필요한 지점은 두 곳이다: (1) 계획서 검토, (2) 커밋·push 직전.** 그 외(PR 생성/갱신, 검토→개발 재작업 루프 진입)는 승인을 묻지 않고 바로 진행한다.
- "애매한 부분 확인"은 승인 요청이 아니라 계획에 없는 요구사항을 채우기 위한 질문이므로 별개로 계속 유지한다.
- PR을 `develop`에 머지하는 것과 `develop`을 `main`으로 승격(배포)하는 것은 파이프라인에 포함되지 않는다 — 항상 사용자가 직접 처리한다.

## 공통 규칙 (모든 서브에이전트)
- 작업을 시작하기 전에 TodoWrite로 할 일 체크리스트를 먼저 만든다.
- 작업을 진행하면서 체크리스트 항목을 하나씩 완료 처리한다. 이 체크리스트가 사용자에게 보이는 진행 상황판 역할을 한다.

## 디렉토리 컨벤션
- 화면: `src/pages/`
- 공용 컴포넌트: `src/components/`
- 계획 문서: `docs/plans/<feature-slug>.md` (템플릿: `docs/plans/TEMPLATE.md`)
- 스타일: Tailwind 유틸리티 우선, 반복되는 조합은 컴포넌트로 추출
- 아이콘: lucide-react. 커스텀 에셋은 사용자가 지정한 경로 사용

## 커밋 컨벤션
Conventional Commits 형식을 사용한다: `feat`, `fix`, `style`, `refactor`, `chore`, `docs`, `test` 등
예: `feat: 홈 화면 카드 리스트 구현`

## 이슈 / PR 템플릿
- 이슈: `.github/ISSUE_TEMPLATE/task.md`
- PR: `.github/pull_request_template.md`
- developer 서브에이전트가 PR을 올릴 때 이 PR 템플릿 형식을 채워서 작성한다.

## 서브에이전트 요약
| 서브에이전트 | 파일 | 할 수 있는 것 | 못 하는 것 |
|---|---|---|---|
| planner (기획) | `.claude/agents/planner.md` | 피그마 분석, `docs/plans/**` 계획 문서 작성 | `src/**` 코드 수정, 커밋 |
| developer (개발) | `.claude/agents/developer.md` | 브랜치 생성, 코드 구현, PR 생성(승인 불필요) | `main`/`develop` 직접 커밋, 승인 없는 커밋·push, PR 머지, 애매한 부분 임의 판단 |
| reviewer (검토) | `.claude/agents/reviewer.md` | 빌드/린트/시각 비교, developer 재호출 | 직접 코드 수정, PR 머지 |
